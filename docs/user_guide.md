# User Guide: FCRAG 2.0 NOC Dashboard

The FCRAG 2.0 dashboard is built with Streamlit to mimic a modern Network Operations Center (NOC) environment. It provides real-time telemetry visualization and a complete interface for triggering and reviewing autonomous Root Cause Analyses.

## Starting the Dashboard

To start the UI, execute the following from the root directory:
```bash
streamlit run src/app.py
```
*Note: Because FCRAG initializes local embedding models and connects to Hugging Face APIs, the initial load may take 10-15 seconds.*

## Understanding the Interface

### 1. Live Telemetry Monitor
The top of the dashboard features three simulated real-time sparkline charts:
- **Signal Strength (dBm)**
- **Latency (ms)**
- **Throughput (Mbps)**

These metrics emulate a live cell tower monitoring feed. When an anomaly drops below threshold (e.g., Latency > 25ms), the color indicators shift from green to amber/red.

### 2. Anomaly Injection Controls
Located just below the telemetry feeds, the control panel offers two ways to interact with the FCRAG Agentic Pipeline:

#### A. Auto-detect from Scenario Library
- Clicking **" Detect Random Network Anomaly"** randomly selects a predefined fault from `data/custom_scenarios/fault_clause_mapping.json`.
- Alternatively, you can click specific preset scenarios (e.g., "⚡ HO_FAILURE: A3 offset too aggressive...").
- This simulates an automated alarm trigger from a downstream monitoring system.

#### B. Manual Fault Description
- You can type free-form text describing a problem, such as: *"We are seeing persistent handover failures on Cell 42, A3 offset seems misconfigured, causing ping-pong effects."*
- Click **" Analyze Manual Fault"** to force the agentic DAG to process your natural language query.

## The Analysis Timeline

Once an anomaly is triggered, a timeline appears tracking the LangGraph agent execution in real-time:
1. **Hybrid Retrieval:** Executes sparse (BM25) and dense (Qdrant) queries.
2. **Cross-Encoder Reranking:** Filters out irrelevant matches.
3. **Context Assembly:** Prepares the 3GPP text.
4. **LLM Reasoning:** Sends the context to `Llama-3.2-3B-Tele-it`.

## Reviewing Results

After the analysis completes, the results are split into three tabs:

###  Dashboard Overview
Displays high-level performance KPIs of the current run:
- **Retrieved Clauses:** Number of 3GPP clauses successfully fetched.
- **Retrieval Time / LLM Time:** Exact latency metrics for the operation.
- **Reranker Score Distribution:** A Plotly bar chart showing the Cross-Encoder confidence scores for the top clauses.
- **Source Type Breakdown:** A pie chart showing whether the evidence came from `3gpp_spec`, `simu5g_narratives`, or `alarm_history`.

###  RCA Report
Displays the finalized diagnostic generated by the LLM, split into distinct sections:
- **⚠ Problem Description**
- **🔍 Root Cause Analysis**
- **✅ Recommendations**
- **💬 Full LLM Response:** A raw, unformatted code-block showing exactly what the LLM returned.
- **Ground Truth Reference:** (If a preset scenario was used), this shows what the system *expected* the fault to be to verify the LLM's accuracy.

###  Evidence
A deep-dive into the raw technical context that guided the LLM:
- Displays all retrieved 3GPP clauses ranked by relevance score.
- You can expand each clause to read the exact, unedited text pulled directly from the specifications (e.g., TS 38.331).
